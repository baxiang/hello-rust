## 17.1 为什么需要生命周期？

### 悬垂引用问题

```rust
// ❌ 错误：返回悬垂引用
// fn create_reference() -> &i32 {
//     let x = 5;      // x 在栈上创建
//     &x              // 返回 x 的引用
// }                   // x 在这里被释放，但引用仍然返回！

// 这会导致：
// • 访问已释放的内存
// • 程序崩溃或未定义行为
// • 安全漏洞
```

### 生命周期的作用

```
┌─────────────────────────────────────────────────────┐
│              生命周期的作用                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  核心目标：防止悬垂引用                              │
│  ├── 确保引用不会比它指向的数据活得更久             │
│  ├── 编译时检查，无运行时开销                       │
│  └── Rust 内存安全的关键机制                        │
│                                                     │
│  与其他语言对比：                                    │
│  ├── C/C++：手动管理，容易出错                      │
│  ├── Java/Python：垃圾回收，有运行时开销            │
│  └── Rust：生命周期检查，零开销且安全               │
│                                                     │
│  实际意义：                                          │
│  • 防止 use-after-free 错误                         │
│  • 防止访问无效内存                                  │
│  • 保证并发安全（与借用规则配合）                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 生命周期与借用规则的关系

```rust
fn main() {
    let mut data = String::from("hello");

    // 不可变借用
    let r1 = &data;
    let r2 = &data;
    // println!("{} {}", r1, r2);  // ✅ 可以

    // data.push_str(" world");  // ❌ 不能：有不可变借用时不能可变借用

    drop(r1);  // r1 生命周期结束

    // 可变借用
    let r3 = &mut data;  // ✅ 现在可以了
    r3.push_str(" world");
}
```
